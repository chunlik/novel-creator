# Status Taxonomy

> 本文件定義 novel-creator repo 允許使用的狀態詞。  
> AI agent 不應隨意創造 `done`, `finished`, `maybe`, `semi_active` 等非標準狀態。  
> 若需要新狀態，先加入本檔，再更新相關腳本 / Dataview / 檢查器。

---

## 核心狀態

| 狀態 | 用途 | 定義 | 可用於 |
|------|------|------|--------|
| `active` | 仍有效 | 已在正文正式建立，後續仍可影響劇情 | 角色、物品、伏筆、系統、規則、場景 |
| `paid` | 已回收 / 已支付 | 伏筆功能已完成，或代價 / 物品已被支付消耗 | 伏筆、物品、交易、角色約束 |
| `completed` | 已完成 | 事件或任務已完成，但不一定代表已消耗 | 任務、角色線、交易 |
| `archived` | 已歸檔 | 副本 / 場景已結束，不能無交代直接重返 | 副本、場景、規則 |
| `planning` | 規劃中 | 已設計但尚未在正文出現 | 新卷大綱、新副本規則、預定機制 |
| `pending_canon` | 待正文落地 | 舊設定或規劃存在，但正式正文尚未明確建立 | 角色限制、物品、伏筆、規則、場景機制 |
| `unknown` | 未知 | 正文沒有足夠資訊判斷 | 位置、身份、動機、狀態 |
| `unresolved` | 未解決 | 已正式建立，但答案尚未揭露 | 伏筆、角色身份、系統問題 |

---

## 物品狀態

| 狀態 | 定義 | 使用規則 |
|------|------|----------|
| `active` | 物品存在且可能影響後續 | 可提及，可使用；需符合持有者與位置 |
| `active_locked` | 物品存在但被限制使用 | 可提及，不可直接使用 / 打開 / 揭露 |
| `paid` | 已消耗 / 已支付 | 不可無交代重新出現 |
| `completed_unresolved_location` | 事件完成但位置未明 | 可作伏筆，不可直接放到主角手上 |
| `unlocked_not_owned` | 商城已解鎖但未持有 | 不可直接使用；需先購買 / 兌換 / 發放 |
| `rejected_hazard` | 被拒收的危險物 | 不可寫成主角持有物 |
| `non_player_item_unresolved` | 非主角物品，位置未明 | 不可無交代出現在主角身上 |
| `coin_active` | 幣類線索仍有效 | 可提及符號 / 編號 / 連貫性 |
| `paid_as_transaction` | 作為交易功能已完成 | 不再重複用同功能，但物品殘留可另行標記 |

---

## 角色狀態

| 狀態 | 定義 | 使用規則 |
|------|------|----------|
| `active` | 角色 / 約束仍有效 | 後續必須遵守 |
| `completed` | 角色功能完成 | 可再出場，但原功能不可重複當新伏筆 |
| `paid` | 角色代價 / 任務已支付 | 不可逆，除非新正文明確改變 |
| `exited` | 已離開當前副本 / 場景 | 不可無交代再出現 |
| `unresolved` | 身份 / 動機未解 | 可暗示，不可定論 |
| `pending_canon` | 舊設定未正式落地 | 需先在正文落地才可視為有效 |

---

## 規則狀態

| 狀態 | 定義 | 使用規則 |
|------|------|----------|
| `active` | 跨章仍有效 | 寫作時必須遵守 |
| `archived` | 副本已結束，規則歸檔 | 不可跨副本直接使用 |
| `archived_case` | 可作案例 / 類比 | 可借鑑，不可直接複製 |
| `paid` | 規則功能已完成 | 可被提及，但不應重複同功能 |
| `planning` | 新副本預定規則 | 必須在正文呈現後才算正式 |
| `pending_canon` | 尚未在正文建立 | 不可直接當成事實 |

---

## 場景狀態

| 狀態 | 定義 | 使用規則 |
|------|------|----------|
| `active` | 當前可用場景 | 可以直接寫入，但要遵守場景機制 |
| `archived` | 已結束場景 | 只能透過回放 / 記憶 / 歸檔 / 重啟使用 |
| `planning` | 預定場景 | 需在正文建立入口和規則 |
| `absent` | 本作品目前不存在 / 不可用 | 不可直接使用 |
| `forbidden` | 明確禁止 | 使用前必須有大事件改變設定 |
| `unknown` | 未知 | 不可假設細節 |

---

## 系統狀態

| 狀態 | 定義 | 使用規則 |
|------|------|----------|
| `active` | 目前有效 | 寫作時必須遵守 |
| `archived` | 系統已結算 / 歸檔 | 不可當前使用，除非重新打開 |
| `rebuilt` | 由腳本重建 | 表示資料根據章節重新整理過 |
| `locked` | 被限制使用 | 不可直接打開 / 使用 / 揭露 |
| `likely_but_unconfirmed` | 合理推定但未確認 | 必須在正文或系統提示確認後使用 |

---

## 禁止使用的非標準狀態

AI agent 不應使用以下狀態詞：

- `done`
- `finished`
- `resolved`
- `closed`
- `maybe`
- `semi_active`
- `probably`
- `inactive`（請改用 `archived`, `paid`, `completed`, `exited`）
- `used`（請改用 `paid`, `completed`, `paid_as_transaction`）

---

## 狀態選擇流程

1. **正文已明確建立，後續仍有效？** → `active`
2. **功能已完成 / 代價已支付？** → `paid` 或 `completed`
3. **副本 / 場景已結束？** → `archived`
4. **只是規劃，正文未出現？** → `planning`
5. **舊設定存在但正式正文未落地？** → `pending_canon`
6. **正文證據不足？** → `unknown`
7. **問題已建立但答案未揭露？** → `unresolved`

---

## AI 寫作規則

- 寫章節時，不可把 `planning` 當成讀者已知規則。
- 寫章節時，不可把 `pending_canon` 當成既成事實。
- 修改 database 時，不可用非標準狀態詞。
- 若章節讓 `pending_canon` 正式落地，需同步更新相關 database，將狀態改為 `active`。
- 若物品被消耗，需同步更新 `物品連續性.md` 為 `paid` 或其他合適狀態。
