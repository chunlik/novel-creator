# AI協作規則

## 文件狀態

【已定案】

本文規範《無限直播間+》後續與 AI 協作時的工作方式。所有 AI 協作、設定整理、資料庫更新與正文創作，均應優先遵守本文。

---

## 1. 討論語言與文件語言分離

【已定案】

### 1.1 討論階段

討論階段可以使用：

- 台灣繁體中文；
- 廣東話；
- 混合口語；
- 臨時腦力激盪語氣。

目的為快速溝通、測試想法、確認方向。

### 1.2 文件階段

所有寫入 World Bible 的正式文件，必須使用：

> 標準台灣繁體中文。

正式文件禁止使用：

- 粵語口語；
- 網路梗；
- 臨時討論語氣；
- 情緒化評語；
- 未標記狀態的推測內容。

---

## 2. 設定狀態標記

【已定案】

正式文件中的設定必須標記狀態。

### 2.1 【已定案】

代表不可輕易修改。若需修改，必須先進行衝突檢查。

### 2.2 【已定案】

代表大方向已確認，但細節仍可調整。

### 2.3 【已定案】

代表仍是候選方案、推測或未完成討論內容。

### 2.4 【廢案】

代表已暫時放棄或不採用，但可保留作備查。

## 4. 設定必須有功能

【已定案】

每個設定都必須回答：

> 它解決什麼問題？

常見功能包括：

- 推動主線；
- 強化角色弧線；
- 支撐世界觀；
- 製造懸念；
- 回收伏筆；
- 製造恐怖感；
- 提供爽點；
- 提供笑點；
- 襯托主角能力。

若設定目前只有「很酷」而無明確功能，應先放入 `99-廢案區`。

---

## 5. 大型設定四層檢查

【已定案】

重要設定正式定案前，必須通過四層檢查：

1. **世界觀檢查**：是否符合既有底層規則？
2. **劇情檢查**：是否推動故事，而不是只增加複雜度？
3. **角色檢查**：是否搶走主角或核心角色功能？
4. **終局檢查**：是否能在後期回收？

四項皆通過後，才可升級為【已定案】。

---

## 6. 保持主角特殊性

【已定案】

新增任何設定時，必須檢查是否削弱周倉的主角地位。

目前原則：

- 0號樣本可以是先驅，但不能成為完成者；
- 林█可以是關鍵線索，但不能替周倉解決終局；
- 小眠可以是鏡像案例，但不能取代許小小情感核心；
- 存續者可以提供資源，但周倉不可成為任何派系的工具人。

---

## 7. 正式寫入前必須衝突檢查

【已定案】

任何會影響 Level A 設定的調整，必須先檢查：

- 是否與已定案設定衝突；
- 是否造成時間線矛盾；
- 是否造成角色功能重疊；
- 是否削弱主角特殊性；
- 是否讓終局無法回收。

若有風險，需先寫入 `世界觀衝突檢查.md`，不得直接改動核心設定。

## 9. AI 回答格式建議

【已定案】

進行 brainstorming 時，優先使用：

- A / B / C / D 選項；
- 每項優點、缺點、影響；
- 推薦方案；
- 定案後應寫入哪個檔案。

完成討論後，正式文件必須改寫為客觀、清晰、可搜尋的台灣繁體中文。

## 10. 禁止跨層污染

【已定案】

設定資料庫採用分層結構，不同層級不得互相混淆。

建議層級：

| 層級 | 類型 | 範例 |
|---|---|---|
| Layer 0 | 核心問題 | `Q001 被刪除後是否仍然存在？` |
| Layer 1 | 世界公理 | `存在校正`、`存續人格` |
| Layer 2 | 世界現象 | `直播間`、`碎片立場理論` |
| Layer 3 | 組織與事件 | `第零修復科`、`歸零事件` |
| Layer 4 | 角色 | `周倉`、`許小小` |
| Layer 5 | 篇章與副本 | `便利店篇`、`失物檔案館篇` |

錯誤做法：

- 將角色設定成世界公理；
- 將組織設定成世界法則；
- 將單一副本設定成整個世界的底層規則。

正確做法：

> 核心問題 → 世界公理 → 世界現象 → 組織／事件 → 角色 → 篇章與副本。

任何新增設定必須先確認其層級。

## 11. 先建立模板，再建立內容

【已定案】

若出現新的文件類型，應先建立模板，再建立正式文件。

標準流程：

1. 判斷新內容類型；
2. 建立或套用對應模板；
3. 填入正式內容；
4. 建立雙向連結；
5. 標記狀態與 Level；
6. 檢查依賴關係；
7. 必要時進行衝突檢查；
8. 寫入 Repo。

禁止大量新增格式不一致的正式設定文件。

---

## 12. 小步提交規則

【已定案】

Repo 是《無限直播間+》的單一真實來源。已定案且低風險的內容，應儘快寫入 Repo，不應長期停留在聊天紀錄中。

### 12.1 應立即寫入的內容

以下內容在定案後應儘快提交：

- 模板；
- 控制台文件；
- 索引文件；
- Obsidian 規範；
- 已通過衝突檢查的 Level B／Level C 設定；
- 不影響主線的整理性文件。

### 12.2 需先凍結與檢查的內容

以下內容不得直接寫入正式定案區：

- 林█；
- 0號樣本；
- X；
- 系統起源；
- 歸零事件完整版；
- 會改變周倉特殊性的設定；
- 會改變許小小核心定位的設定。

此類內容必須先寫入 `待決議題.md` 或 `世界觀衝突檢查.md`。

### 12.3 提交節奏

推薦流程：

```text
討論 → 定案 → 衝突檢查 → 寫入 Repo → 更新變更日誌 → 繼續討論
```

禁止長期累積大量已定案內容後一次性寫入，避免遺漏、版本偏差與設定漂移。
