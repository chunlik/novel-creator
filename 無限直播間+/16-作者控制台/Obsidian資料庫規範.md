# Obsidian資料庫規範

## 文件狀態

【已定案】

本文規範《無限直播間+》World Bible 在 Obsidian 中的使用方式。此資料庫不只作為資料夾集合，而應作為可視化、可搜尋、可連結的小說知識圖譜。

---

## 1. 核心原則

【已定案】

《無限直播間+》資料庫採用 Obsidian-first 設計。

所有正式設定文件應優先考慮：

- 雙向連結；
- Tags；
- Hub 索引；
- Dataview 查詢；
- 依賴關係；
- 衝突檢查；
- Graph View 可讀性。

資料夾只負責基本分類，真正的知識結構應由雙向連結與 Hub 文件建立。

---

## 2. 雙向連結規則

【已定案】

所有重要設定首次出現時，應使用 Obsidian 雙向連結。

例如：

```markdown
[[周倉]]
[[許小小]]
[[存在校正]]
[[存續人格]]
[[歸零事件]]
```

### 2.1 必須建立連結的類型

以下內容必須建立獨立頁面，並使用雙向連結：

- 主要角色；
- 核心概念；
- 核心事件；
- 組織與派系；
- 重要副本；
- 核心問題；
- 重要伏筆；
- 篇章；
- 終局設定。

### 2.2 連結目的

雙向連結的目的不是裝飾，而是讓 AI 與作者能快速理解：

- 哪些設定互相依賴；
- 哪些角色與同一謎團有關；
- 哪些概念支撐同一篇章；
- 修改某一設定會影響哪些文件。

---

## 3. Tags 規則

【已定案】

Tag 必須克制使用，禁止為每個細節都建立 Tag。

### 3.1 標準 Tag

正式文件可使用以下標準 Tag：

```markdown
#角色
#組織
#事件
#概念
#問題
#副本
#篇章
#伏筆
#終局
#規則
#素材
#笑點
#避坑
#控制台
```

### 3.2 狀態 Tag

可使用以下狀態 Tag：

```markdown
#已定案
#部分定案
#待定
#凍結
#廢案
```

### 3.3 等級 Tag

可使用以下重要度 Tag：

```markdown
#Level-A
#Level-B
#Level-C
```

### 3.4 禁止事項

禁止使用過度細碎的 Tag，例如：

```markdown
#很酷
#之後再說
#可能有用
#我喜歡
```

此類內容應寫入正文說明或放入 `99-廢案區`。

## 4. Hub 文件規則

【已定案】

每個大型分類應建立 Hub 文件。

Hub 文件不是詳細設定，而是入口與導航。

建議 Hub：

```text
00-總覽/世界觀總索引.md
00-總覽/核心問題總表.md
00-總覽/角色總索引.md
00-總覽/組織總索引.md
00-總覽/副本總索引.md
00-總覽/篇章總索引.md
```

Hub 文件應包含：

- 重要頁面連結；
- 狀態標記；
- 依賴關係摘要；
- 待補項目；
- 與核心問題的關聯。

---

## 5. 核心問題節點

【已定案】

《無限直播間+》的 Graph View 應以核心問題作為主要中心節點之一，而不只以角色作為中心。

核心問題文件建議放置於：

```text
00-總覽/核心問題/
```

範例：

```text
Q001-一個人被刪除後是否仍然存在.md
Q002-修復是否等於抹殺.md
Q003-世界是否只能有一個正確答案.md
```

所有角色、概念、事件文件，若與核心問題有關，應反向連結至對應問題。

---

## 6. 依賴關係區塊

【已定案】

Level A 與 Level B 文件必須包含「依賴關係」區塊。

建議格式：

```markdown
## 依賴關係

### 本設定依賴

- [[存在校正]]
- [[存續人格]]

### 被以下設定依賴

- [[歸零事件]]
- [[0號樣本]]

### 修改風險

【高】修改此設定會影響主線、終局或核心角色。
```

此區塊用於防止設定修改時產生連鎖衝突。

## 7. YAML Frontmatter 建議

【已定案】

正式文件建議加入 YAML Frontmatter，方便 Dataview 查詢。

範例：

```yaml
---
type: 角色
status: 已定案
level: Level-A
tags:
  - 角色
  - 已定案
  - Level-A
related:
  - 周倉
  - 許小小
  - 存續人格
updated: 2026-05-31
---
```

### 7.1 建議欄位

| 欄位 | 用途 |
|---|---|
| `type` | 角色、概念、事件、組織、副本、篇章、伏筆等 |
| `status` | 已定案、部分定案、待定、凍結、廢案 |
| `level` | Level-A、Level-B、Level-C |
| `tags` | Obsidian 標籤 |
| `related` | 重要關聯節點 |
| `updated` | 最後更新日期 |

## 8. Dataview 使用建議

【已定案】

若使用 Dataview，可建立以下查詢頁：

### 8.1 已定案 Level A 設定

```dataview
TABLE type, status, level
FROM "無限直播間+"
WHERE status = "已定案" AND level = "Level-A"
SORT file.name ASC
```

### 8.2 待定設定

```dataview
TABLE type, level
FROM "無限直播間+"
WHERE status = "待定"
SORT file.name ASC
```

### 8.3 凍結設定

```dataview
TABLE type, related
FROM "無限直播間+"
WHERE status = "凍結"
SORT file.name ASC
```

### 8.4 角色清單

```dataview
TABLE status, level, related
FROM "無限直播間+"
WHERE type = "角色"
SORT file.name ASC
```

## 11. Graph View 使用目標

【已定案】

Graph View 的主要用途不是美觀，而是檢查：

- 是否有孤立設定；
- 哪些設定過度中心化；
- 哪些角色與謎團連線過多；
- 哪些核心問題支撐最多設定；
- 哪些待定設定可能造成衝突。

若某個新設定完全沒有連到任何核心問題、角色、事件或篇章，必須重新評估其必要性。

---

## 12. 寫入正式文件的最低要求

【已定案】

任何正式設定文件至少應包含：

1. YAML Frontmatter；
2. 文件狀態；
3. 核心定義；
4. 功能；
5. 關聯連結；
6. 依賴關係；
7. 修改風險；
8. 待定問題。

Level C 素材類文件可簡化，但仍需標明狀態與用途。
